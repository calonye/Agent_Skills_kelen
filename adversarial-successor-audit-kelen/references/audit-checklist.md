# 审计清单 / Audit Checklist

> 逐步模拟新人接手流程，每一步用下列判据检查。

## 阶段 1：Clone + 首次浏览

- [ ] README 是否在前 20 行内说明项目是什么、如何开始
- [ ] 是否有明确的「快速开始」步骤（clone → install → setup → dev）
- [ ] 引用的链接是否全部有效（不指向不存在的文件/目录）
- [ ] 是否说明了仓库层级关系（fork/上游/源头，如适用）
- [ ] 许可证是否存在且正确

## 阶段 2：安装依赖 + 初始化

- [ ] `<package-manager> install` 能否一次成功（锁文件与 package.json 一致）
- [ ] `setup` 命令是否存在且能跑通（不依赖 git-ignored 文件 — **Bootstrap 死循环检测**）
- [ ] setup 后 git remote 是否正确配置
- [ ] setup 后 git hooks 是否安装
- [ ] 必要的环境变量 / 配置文件是否有模板或自动生成

## 阶段 3：日常开发

- [ ] 是否有明确的分支策略文档（哪个分支开发、哪个分支稳定）
- [ ] 切分支命令是否可用（不报错、不依赖缺失文件）
- [ ] 是否有「开工前先同步上游」的提醒或自动检查
- [ ] dev server 能否正常启动（`bun run dev` / `npm run dev`）

## 阶段 4：提交

- [ ] 提交规范是否有文档说明（commit message 格式）
- [ ] git hooks 是否自动校验提交格式
- [ ] 是否阻止了向主干直接 commit
- [ ] type-check / lint 是否在提交前自动运行

## 阶段 5：同步上游

- [ ] 同步命令是否存在且可用
- [ ] 是否交互式选择同步源（且支持非交互参数 — **AI 可调用性检验**）
- [ ] 冲突时是否有清晰的中文错误提示
- [ ] 同步后是否自动合回开发分支

## 阶段 6：合并到主干

- [ ] 是否有预检步骤（typecheck / lint / build）
- [ ] 是否有 PR 判断流程（哪些改动应回馈上游）
- [ ] push 前是否有二次确认
- [ ] 是否阻止了向非 origin 远端 push

## 阶段 7：发布

- [ ] tag 命名是否有文档/自动化
- [ ] CI 是否能被正确触发
- [ ] release 产物是否正确

## 横切检查

### Bootstrap 死循环

- [ ] 所有 `package.json` scripts 指向的文件都存在于仓库中（非 git-ignored）
- [ ] `setup.sh` / `install.sh` 等初始化脚本本身入库
- [ ] 初始化过程中不依赖初始化的输出（循环依赖）

### 入库边界

- [ ] 每个文件用三层判据检查：客观规范 → 入库 / 自动化工具 → 入库 / 个人偏好 → 不入库
- [ ] `.gitignore` 注释与实际行为一致（不说谎）
- [ ] 没有应入库的文件被忽略，没有应忽略的文件被追踪

### AI 可调用性

- [ ] 所有交互式脚本（`read` / `select`）支持 `--yes` / `--type=` 等非交互参数
- [ ] 脚本失败时返回非零退出码 + 有意义的错误信息
- [ ] 脚本不依赖特定终端特性（如颜色仅做增强，非必需）
