# 第一性原理判据链 / First Principles Decision Chain

## 定义

当面对「要不要做某事」「选 A 还是选 B」「这个值不值得」等决策时，不靠直觉或惯例，而是回到最基本的判据链条逐层推导。

## 通用判据链（4 层）

### 第 1 层：必要性

> 「这件事不做会怎样？」

- 不做会出错/阻塞 → 必要，继续
- 不做只是不够完美 → 非必要，标记为可选
- 不做完全没影响 → 不做

### 第 2 层：唯一性

> 「是否已有等价的东西？」

- 无等价物 → 唯一，继续
- 有等价物但有本质区别（方向/场景/输入输出不同）→ 非重叠，继续
- 有等价物且差异 < 20% → 重叠，应该迭代现有物而非新建

### 第 3 层：可结构化

> 「能否编码为可复用的流程？」

- 有明确的输入/输出/步骤 → 可结构化
- 是纯主观判断无法分解 → 不可结构化，作为经验记录而非 skill
- 部分可结构化 → 结构化核心流程，主观部分作为参考

### 第 4 层：投入产出比

> 「做这件事的成本 vs 收益？」

- 轻量级（< 1 小时，< 5 个文件）→ 直接做
- 中量级（1-4 小时）→ 需要有 ≥ 3 次预期复用才值得
- 重量级（> 4 小时）→ 需要论证长期价值

## 领域特化判据

### 对于「是否入库」判据

```
客观项目规范（身份/分支/提交/工具链）→ 入库
自动化工具（脚本/钩子/CI）→ 入库
个人偏好/隐私（人格/记忆/私有路径）→ 不入库
```

### 对于「是否转化为 skill」判据

```
可复用频率 > 1 次 → 值得
与现有 skill 有本质区别 → 值得
有明确的输入/流程/输出 → 值得
以上全满足 → 转化
```

### 对于「是否向上游提 PR」判据

```
改动是通用 bug 修复/性能优化/通用新功能 → 提 PR
改动仅适用于本 fork（链接/身份/个人化）→ 不提
依赖升级 → 视上游是否已自行升级
```

## 使用方式

在辩证自我审查的步骤 E（裁定）中，对每个争议点使用此判据链：

```
论点 X: <假设/决策>
→ 第 1 层 必要性: <不做会怎样？>
→ 第 2 层 唯一性: <有等价物吗？>
→ 第 3 层 可结构化: <能编码为流程吗？>
→ 第 4 层 投入产出比: <成本 vs 收益？>
→ 裁定: 成立 / 推翻 / 修正
```
