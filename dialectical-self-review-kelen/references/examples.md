# 实际案例：辩证自我审查的三轮迭代

## 案例 1：全私有方案 vs 骨架入库

### 当前方案
所有规则文件全部放入 git-ignored 目录。

### 论点提取
1. 假设：AI 工具会通过 setup 命令生成入口文件
2. 假设：换机器后 setup 命令能恢复一切
3. 假设：用户说「不入库」= 所有非源码都不入库

### 辩证过程

**论点 1：AI 工具会运行 setup**
- 反驳：AI 工具不会主动看 README 的第 36 行然后跑 shell 脚本。它们读 AGENTS.md / CLAUDE.md — 但这些文件不存在。
- 反反驳：可以在全局规则里告诉 AI 先跑 setup... 但全局规则里不知道这个项目需要 setup。
- 裁定：**推翻**。AI 入口文件必须在仓库里存在。

**论点 2：setup 能恢复一切**
- 反驳：setup 脚本本身在 git-ignored 目录中。clone 后不存在。`bun run setup` → `No such file`。
- 反反驳：（无法辩护 — 这是 Bootstrap 死循环。）
- 裁定：**推翻**。setup 脚本本身必须入库。

**论点 3：用户「不入库」= 全部不入库**
- 反驳：关键词是「非源码直接相关」。AGENTS.md 的分支规则、提交规范直接关系到源码如何被修改，属于「源码直接相关」。
- 裁定：**修正**。拆分为两层 — 项目级规范入库，个人偏好不入库。

### 修正后方案
两层架构：骨架入库 + 个人层私有。

## 案例 2：「差不多了」→ 再审一遍

### 辩证过程

**论点：改动已完成，可以提交**
- 反驳：git-ignored 目录里有废弃残留。新人会困惑「哪个是真的」。
- 裁定：**修正**。清理残留，只保留有效文件。

**论点：.gitignore 注释没问题**
- 反驳：注释说「自动化脚本本地私有」但脚本已经入库了。
- 裁定：**推翻**。更新注释。

### 价值
「差不多了」后再审一轮，总能发现新问题。辩证审查的核心：**收敛不是一次性的，而是迭代式的。**

## 案例 3：交互脚本的 AI 可调用性

### 辩证过程

**论点：交互式脚本已完成**
- 反驳：AI 工具执行脚本时没有 stdin。`read` 会阻塞或直接失败。
- 反反驳：AI 工具可以用管道传入参数... `echo "1" | bash sync.sh`...
- 反反反驳：这太 hack 了。如果选项变了怎么办？
- 裁定：**修正**。所有脚本加 `--yes` 等非交互参数模式。

### 启示
「技术上可行 ≠ 合理可用」。反反驳阶段承认辩护不住（hack）是关键的诚实条款。
