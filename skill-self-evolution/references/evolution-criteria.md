# 进化判据详解 / Evolution Criteria

## 4 层验证的边界案例

### 第 1 层 必要性 — 边界情况

**通过的例子：**
- 「辩证自我审查」— 不固化 → 下次遇到复杂决策时 AI 又会直觉行事，不会自我反驳
- 「对抗性接替者审计」— 不固化 → 下次交付前不会主动做新人模拟

**不通过的例子：**
- 「这个项目用 bun 而非 npm」— 这是项目配置，不是方法论，记在 AGENTS.md 即可
- 「用户喜欢中文为主」— 这是个人偏好，记在 private-rules.md 即可

**判断技巧：** 问「换一个完全不同的项目，这个方法论还有用吗？」如果是 → 必要。如果只在当前项目有用 → 不是方法论，是项目规则。

### 第 2 层 唯一性 — 如何判断重叠

对比维度表：

| 维度 | 若不同则非重叠 |
|------|---------------|
| 输入 | 当前方案 vs 历史 JSONL vs 代码 diff |
| 输出 | 修正方案 vs brain 笔记 vs skill 包 vs 审计报告 |
| 时机 | 决策前 vs 事后 vs 实时 |
| 方向 | 内向（审查自己）vs 外向（审查系统/代码） |
| 模式 | 收敛（反驳→裁定）vs 发散（探索→选择） |

如果 3 个以上维度不同 → 非重叠，可创建新 skill。
如果 1-2 个维度不同 → 考虑迭代现有 skill 而非新建。
如果 0 个维度不同 → 重叠，不创建。

### 第 3 层 可结构化 — 什么不能结构化

不能结构化为 skill 的：
- 「要有品味」— 纯主观，无法分解为步骤
- 「注意安全」— 太泛，没有具体的检查点
- 「写好代码」— 没有明确的输入/输出

可以结构化的标志：
- 有触发条件（什么时候用）
- 有步骤序列（按什么顺序做）
- 有输出格式（产出什么）
- 有验证方式（怎么知道做对了）

### 第 4 层 投入产出比 — 特殊情况

用户明确要求「把这个变成 skill」→ 跳过投入产出比判断，直接执行。
（用户意愿 > 投入产出比计算。但仍需通过前 3 层。）

## 转化 vs 迭代的决策树

```
候选方法论 M
  ├─ 与现有 skill S 重叠 ≥ 3 维度 → 不做
  ├─ 与现有 skill S 重叠 1-2 维度 → 迭代 S
  │   ├─ 新步骤？→ 加入 S 的流程
  │   ├─ 新参考？→ 加入 S 的 references/
  │   └─ 新案例？→ 加入 S 的 examples.md
  └─ 与所有现有 skill 重叠 0 维度 → 创建新 skill
```
