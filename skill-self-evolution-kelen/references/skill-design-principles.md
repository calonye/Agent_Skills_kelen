# Skill 设计原则 / Skill Design Principles

> 本文件是 skill-self-evolution 转化新 skill 时的强制参考。无论在什么 AI 工具或环境中，
> 产出的 skill 都必须遵循以下原则。

## 格式规范

- SKILL.md 主文件 ≤ 500 行，长内容拆到 `references/`
- 列表项间无空行，同一文件内连续空行不超过 1 个
- 符号替代冗余词（`→` 代替「然后」「流转到」）
- 禁用表格，优先使用自然语言列表
- 避免非必要加粗/特殊字符/空行
- 优先级：规则目的（逻辑完整性）> 模型理解效率 > Token 效率 > 人类可读性

## Frontmatter 规范

```yaml
---
name: <skill-name>-kelen
description: >-
  <功能描述>。
  触发场景（用户口语）：
  「<口语触发词1>」「<口语触发词2>」
  触发场景（AI 自我思考）：
  「<推理触发模式1>」「<推理触发模式2>」
  关键词：<关键词1>、<关键词2>。
metadata:
  author: kelen
  version: <版本号>
---
```

必填字段：`name`、`description`、`metadata.author`、`metadata.version`

## 触发词覆盖

### 两层路由机制

Skill 触发依赖两层协作，缺一不可：

- **发现层（description 字段）**：当 AI 已经决定要搜索 skill 时，description 字段决定语义匹配结果。写得再好，如果触发层没有启动搜索动作，description 永远不会被读取。
- **触发层（CLAUDE.md / AGENTS.md 路由规则）**：决定 AI 是否启动 skill 搜索。上下文路由（Contextual Routing）是触发层的核心——基于对话状态主动触发，而非依赖用户显式指令。

两层断裂的典型症状：skill 文件存在且 description 完善，但整个对话过程中从未被调用。根因通常是触发层缺失或规则中的名称与 skill 目录名不一致。

### 内容要求

- 必须同时覆盖「用户口语触发」和「AI 推理触发」两个维度
- 口语触发：模拟用户日常表达，如「帮我检查一下」「先想清楚再做」
- 推理触发：AI 在推理过程中识别到需要使用此 skill 的场景
- 目标：不仅用户明确发出指令时触发，AI 自身推理时也能因语义匹配自动触发

## 内容净化

### 禁止出现

- **心路历程**：「从 XX 对话中提炼」「这个设计是我在做 XX 的时候想到的」
- **非权威引用**：「遵循 XX 规范」（除非 XX 已是行业标准）
- **维护说明**：「Claude 侧维护」「Factory 侧维护」「按需读取」
- **无实际规则意义的描述**：仅描述「从哪来」「谁维护」的注释

### 允许保留

- 实际规则和约束条件
- 流程步骤和输出格式
- 触发条件和关键词
- 与其他 skill 的功能边界说明（如「核心区别」对比段落）

### 权重判断

引用/参考/灵感来源 → 低权重。若经用户确认需要保留：
- 放在文件末尾或脚注位置
- 作为注释类型出现
- 不出现在 description 或核心流程段落中

## 明确判断不干扰原则

- **高置信度判断**（如移除明显的心路历程）→ 直接执行，不征求用户确认
- **模糊/有争议判断**（如某个引用是否属于行业标准）→ 必须征求用户意见
- 目的：避免上下文被确认环节污染，同时保留关键决策的用户控制权

## 规则与说明分离

- **SKILL.md**：给 AI 执行的规则和流程
- **README.md**（如有）：给人类阅读的说明、设计背景、版本历史
- 两者内容不混放

## 目录结构

```
<skill-name>-kelen/
├── SKILL.md              # 主文件（路由 + 流程骨架，≤ 500 行）
├── agents/
│   └── interface.yaml    # 接口声明（可选）
├── references/           # 详解、判据、案例（按需加载）
│   ├── <topic>.md
│   └── adapters/         # 适配案例（环境相关时使用）
└── README.md             # 人类说明（可选，不给 AI 读取）
```

## 渐进式披露

- **第 1 层**（metadata）：name + description，始终在上下文中（~100 tokens）
- **第 2 层**（SKILL.md body）：skill 触发时加载（≤ 500 行）
- **第 3 层**（references/）：按需加载（无限制）

## 验证原则

Skill 设计和部署后必须验证，否则等于没做：

1. **改了规则 ≠ 改了行为**：规则写进文件后，必须在新环境/session 中验证行为是否变化，否则等于没改。
2. **同步后必须做一致性检查**：自动同步脚本跑过了不等于同步成功。同步后必须 checksum 比对两端内容一致。
3. **命名变更后做全链条搜索一致性验证**：重命名 skill 后，从 `name` 字段 → 目录名 → interface.yaml → 路由规则 → Readme 触发词，全链条必须一致。用 `rg` 搜索旧名确认无遗漏。

## 交叉引用规范

每个 skill 的 SKILL.md 应包含与其他 skill 的功能边界和交叉触发关系：

- 在核心流程中标注「交叉引用」：当某个步骤可以或应该触发另一个 skill 时，明确写出来
- 在「不做什么」段落中标注功能边界：明确本 skill 不做什么、该用哪个 skill
- 三个核心 skill 形成闭环：`skill-self-evolution`（提炼）→ `dialectical-self-review`（思辨验证）→ `adversarial-successor-audit`（交付审计）
