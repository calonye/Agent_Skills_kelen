# 技能自我进化 / Skill Self-Evolution

每次深度对话都可能产生新的方法论。不要让它们随 session 消亡。

**核心区别：**
- `ruminate`：事后挖掘历史对话 JSONL → 产出 brain 笔记
- `brain`：被动记录知识点 → Obsidian vault
- `skill-creator` / `yao-meta-skill`：通用 skill 创建工具 → 从需求到 skill
- **本 skill**：**在当前对话中**实时识别新掌握的方法论 → 判断 → 转化/迭代为可部署 skill 包

**硬约束：必须通过第一性原理判据链（4 层）验证后才转化，不是什么都值得变成 skill。**

## 何时使用

- 深度对话接近尾声，准备提交/收工前
- 用户说「思辨你学到了什么」「这次对话有什么可以变成技能的」
- 多轮迭代中发现了一个反复使用的模式（如同一个判断框架被用了 3 次以上）
- 用户明确教导了一种思维方式并要求固化

## 四步自我进化协议

```
识别方法论 → 第一性原理判断 → 转化/迭代 → 部署验证
     ①              ②              ③           ④
```

### 步骤 ① 识别：从对话中提炼方法论

回顾当前对话（不是历史 JSONL），问自己：

- **思维层面**：我在这次对话中是否使用了一种之前不具备的思考方式？
- **推理层面**：我是否学到了一种新的判断/决策框架？
- **方法层面**：我是否形成了一个可重复的工作流程？
- **逻辑层面**：我是否发现了一种新的论证结构或错误检测模式？

**提取模板：**
```markdown
### 候选方法论 N: <名称>
- 来源：对话中的哪个环节
- 定义：一句话描述
- 输入：什么情况下触发
- 输出：产出什么
- 流程：几个步骤
- 本次使用次数：N 次
```

### 步骤 ② 判断：第一性原理 4 层验证

对每个候选方法论逐层检验（详见 `references/evolution-criteria.md`）：

| 层 | 问题 | 通过标准 |
|----|------|---------|
| 必要性 | 不固化会怎样？ | 会随 session 消亡且未来大概率再次需要 |
| 唯一性 | 现有 skill 能覆盖吗？ | 与现有 skill 有本质区别（方向/输入/输出不同） |
| 可结构化 | 能写成流程吗？ | 有明确的步骤、输入、输出 |
| 投入产出比 | 值得花时间吗？ | 预期复用 ≥ 3 次，或用户明确要求 |

**4 层全通过 → 转化为新 skill 或迭代现有 skill**
**某层不通过 → 记为 brain 笔记或直接跳过（注明原因）**

### 步骤 ③ 转化：创建或迭代（三层决策，必须等用户回复）

**硬约束：每一层呈现选项后必须等待用户回复，不可自行跳过或决定。**
**格式要求：每个选项必须附带快速决策信息（用户一眼看出每个选项意味着什么）。**

#### 第一层：粗粒度决策

对每个通过 4 层验证的方法论，呈现选项 + 快速决策信息：

```markdown
### 候选方法论: <名称>
- 定义：<一句话描述>
- 影响范围：<涉及哪些现有 skill / 新增什么能力>

[1] 独立转化 → 创建新 skill `<name>-kelen`，预计产出 SKILL.md + references/ + interface.yaml
[2] 合入现有 → <推荐合入哪个 skill + 一句话理由，如「与 env-sync-maintainer 职责重叠 70%」>
[3] 跳过 → <跳过原因，如「复用频率预估 < 3 次」>
[4] <AI 思辨建议 → 针对此方法论的具体建议，如「建议拆分为 2 个子 skill」或「建议先做 POC 验证」>
[N] 补充说明（请描述你的想法）
```

⚠️ 必须等待用户回复后再进入下一层。

#### 第二层：细化决策（用户选 [1] 后进入）

- skill 名称建议（-kelen 后缀）
- description 荧稿（口语触发 + AI 推理触发）
- 目录结构建议
- 必须读取 `references/skill-design-principles.md` 确保规范
→ 呈现给用户确认，等待回复

用户选 [2] 后：列出候选 skill + 各自核心职责 + 建议合入理由 → 用户选择后等待回复

用户可在任意一轮回到第一层重新选择。

#### 第三层：确认执行

将最终方案（含文件路径、变更内容、版本号）呈现给用户确认后再写入文件。

#### 路径 1：创建新 skill

1. 在工作空间创建目录：`Agent_Skills_kelen/<skill-name>-kelen/`
2. 按以下结构组织：
   - `SKILL.md`：路由 + 流程骨架
     - frontmatter 必含 `name`、`description`（触发场景）、`metadata`（author/version）
     - description 字段精确描述触发场景（路由匹配核心），包含用户口语触发词 + AI 自我思考触发词
     - 流程步骤保持最小化，详细内容引用 `references/`
   - `agents/interface.yaml`：接口声明
   - `references/`：详解、判据、案例
3. **必须读取 `references/skill-design-principles.md`**，确保新 skill 遵循格式规范和内容净化原则

#### 路径 2：迭代现有 skill

1. 读取现有 skill 的 SKILL.md
2. 识别增量：新流程步骤？新参考文档？新案例？
3. 最小化修改，不破坏现有结构
4. 版本号 patch +1

#### 路径 3：跳过

记录跳过原因，不产出任何文件。

#### 路径 4+：AI 思辨建议 / 补充说明

- AI 思辨建议：针对此方法论的具体建议（如「拆分为 2 个独立 skill」「先做 POC 验证」）
- 补充说明：等待用户输入后，重新进入第二层细化决策

#### 中断续接机制

- 每轮决策结果记入对话上下文
- 若对话中断或切换话题，下次触发 skill-self-evolution 时重新执行步骤 ① 识别 + ② 判断
- 若 session 重启，依赖用户重新触发，方法论不会丢失（已通过 4 层验证的记录在对话中）

### 步骤 ④ 部署验证

1. 将 skill 目录同步到当前 AI 工具的 skill 目录
2. 创建同步脚本（源指向工作空间）
3. 运行同步脚本验证部署成功
4. 确认注册表（update-all 脚本）能扫描到新 skill
5. 更新工作空间 `Readme.md`
6. **名称一致性检查**：确认路由规则（CLAUDE.md / AGENTS.md）中的 skill 名称与工作空间目录名、SKILL.md 中的 name 字段三者一致。不一致会导致触发层断裂——skill 存在但永远不被调用。

## 输出格式

```markdown
## 自我进化报告

### 已识别的方法论
1. <名称> — <一句话定义>
2. ...

### 第一性原理判断
- 方法论 A：必要性 ✓ / 唯一性 ✓ / 可结构化 ✓ / 投入产出比 ✓ → 转化
- 方法论 B：必要性 ✓ / 唯一性 ✗（与 skill-X 重叠 70%）→ 迭代现有 skill
- 方法论 C：必要性 ✗（项目规则，非方法论）→ 跳过

### 执行结果
- 新建 skill: <name> → 已部署到当前环境
- 迭代 skill: <name> → 版本 x.y.z → x.y.z+1
- 跳过: <name> — 原因：<简述>

### 待观察
- <方法论名称> — 暂不转化，原因：<简述>
  - 若后续复用 ≥ 3 次，建议重新触发评估

### 工作空间变更
- 新增文件: ...
- 修改文件: ...
```

## 不做什么

- 不挖掘历史对话文件（那是 `ruminate`）
- 不做通用的 skill 创建教学（那是 `yao-meta-skill`）
- 不把一切都变成 skill — 4 层验证不通过就不转化
- 不修改非 skill 相关的项目文件
- 不依赖特定工具的笔记系统 — 「待观察」方法论记录在输出报告中，而非工具特定的笔记目录
